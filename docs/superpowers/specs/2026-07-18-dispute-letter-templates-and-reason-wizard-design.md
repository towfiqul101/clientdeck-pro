# Agency Static Templates, Letter-Source Choice & Dispute Reason/Instruction Wizard

**Date:** 2026-07-18
**Status:** Approved — ready for implementation planning

## Problem

Today every dispute letter is generated by Claude from an AI-prompt template
selected automatically via `findBestTemplate()`. Agencies want two more
capabilities:

1. Upload their own pre-written letter bodies ("agency static templates")
   and use them as a variable-fill alternative to AI generation, chosen
   per letter.
2. Pick a standard dispute reason (not mine, inaccurate, duplicate,
   obsolete, identity theft, etc.) and instruction (delete/correct/verify)
   per item, and have that injected into whichever letter path is used.

## Data Model

### `letter_templates` — new column

```sql
ALTER TABLE letter_templates
  ADD COLUMN kind TEXT NOT NULL DEFAULT 'ai_prompt'
    CHECK (kind IN ('ai_prompt', 'agency_static'));
```

The existing `prompt_template` column is reused for both kinds. For
`ai_prompt` it's AI instructions (unchanged). For `agency_static` it's the
literal letter body with `{{variable}}` placeholders, filled directly with
no AI call. Existing rows backfill to `kind='ai_prompt'` via the column
default.

### Two new tables

```sql
CREATE TABLE dispute_reasons (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  agency_id UUID REFERENCES agencies(id) ON DELETE CASCADE, -- NULL = system default
  label TEXT NOT NULL,
  is_system BOOLEAN DEFAULT false,
  is_active BOOLEAN DEFAULT true,
  sort_order INTEGER DEFAULT 0,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE dispute_instructions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  agency_id UUID REFERENCES agencies(id) ON DELETE CASCADE,
  label TEXT NOT NULL,
  is_system BOOLEAN DEFAULT false,
  is_active BOOLEAN DEFAULT true,
  sort_order INTEGER DEFAULT 0,
  created_at TIMESTAMPTZ DEFAULT NOW()
);
```

Kept as two separate tables (not one table with a `kind` discriminator) for
clean FK typing on `disputes` and to avoid a cross-kind validity check that
plain `CHECK` constraints can't express.

**Seed data** (`is_system = true`, `agency_id = NULL`):
- Reasons: Inaccurate / Not Mine, Not Mine, Duplicate Account, Obsolete
  (Past Reporting Period), Identity Theft / Fraudulent Account, Incorrect
  Balance, Incorrect Payment History.
- Instructions: Delete, Correct, Verify.

Agencies can add their own custom rows (label only) via the new Dispute
Reasons tab (Section 3); system rows can't be edited or deleted, same
pattern as `letter_templates`.

### `disputes` — three new columns

```sql
ALTER TABLE disputes
  ADD COLUMN dispute_reason_id UUID REFERENCES dispute_reasons(id) ON DELETE SET NULL,
  ADD COLUMN dispute_instruction_id UUID REFERENCES dispute_instructions(id) ON DELETE SET NULL,
  ADD COLUMN letter_source TEXT NOT NULL DEFAULT 'ai'
    CHECK (letter_source IN ('ai', 'agency_template'));
```

`ON DELETE SET NULL` rather than `RESTRICT`: deleting a custom
reason/instruction later must not retroactively break historical disputes
whose letter already baked the label into its content.

This migration will be written as `supabase/migrations/038_...sql` during
implementation. **The user runs it manually in the Supabase SQL editor** —
this environment has no working Supabase MCP/CLI access to this project.

## Templates Settings (`/templates`)

- `template-form.tsx` gets a "Kind" selector (AI Prompt / Agency Template,
  default AI Prompt). Switching it relabels the body field ("Prompt" →
  "Letter Body"); same textarea and the same `{{variable}}` insert-buttons
  panel, which now also includes `{{dispute_reason}}` and `{{instruction}}`.
- Preview button: for `agency_static` templates, runs the deterministic
  local fill (no Claude call, works without `ANTHROPIC_API_KEY`) instead of
  the AI path.
- `templates-list.tsx` cards get a kind badge next to the existing
  System/Custom badge. No new filter tab.
- `findBestTemplate()` gains a `kind` param (default `'ai_prompt'`, so
  existing callers are unaffected); the existing 5-step
  agency→system/exact→generic priority cascade is otherwise unchanged, just
  scoped per kind.

## Dispute Reasons & Instructions Management (`/templates`, second tab)

New tab "Dispute Reasons" alongside "Templates" (same page, same domain —
not a new settings sub-route). Shows two lists side by side (Reasons,
Instructions):
- All active rows (system + this agency's own), system rows marked
  read-only.
- Inline "+ Add custom reason/instruction" (label only).
- Delete on the agency's own custom rows only.

No in-place edit — delete-and-recreate only, matching how custom
`letter_templates` are handled today.

## Round Creation UX (`round-builder.tsx`)

Each selected item's row gains, alongside the existing Letter Type select,
a compact `grid grid-cols-2 sm:grid-cols-3` block (matching the
`template-form.tsx` grid style) with:
- **Reason** select — defaults to "Inaccurate / Not Mine"
- **Instruction** select — defaults to "Delete"
- **Source** toggle — "Generate with AI" / "Use our template", defaults to
  AI (preserves current behavior for agencies not using static templates)

`RoundItemSelection` gains `reasonId`, `instructionId`, `useTemplate:
boolean`. `createRound()` writes these directly onto the new `disputes`
columns.

## Generation Pipeline & Compliance

**`generate-letter.ts`**:
- `GenerateLetterParams` gains `reasonLabel?: string` and
  `instructionLabel?: string`.
- `buildPromptVariables()` adds `dispute_reason` and `instruction` to the
  injected variable set.
- New export `fillTemplateLetter(params)`: builds the same variables, runs
  `injectVariables(template.prompt_template, variables)` with no API call,
  and computes compliance with the near-identical-to-template check
  skipped (that check exists to catch AI generation failures; a
  variable-fill letter is *always* near-identical to its source template by
  design, so it's a structural false positive for this path — see below).
- `validateLetterCompliance()` gains an optional
  `{ skipNearIdenticalCheck?: boolean }` param. When set, that check is
  omitted from the returned `checks` array entirely rather than run and
  ignored. The other three checks (missing citation, unresolved
  `{{placeholder}}`, too-short) still run for both paths — they remain
  meaningful for a template-fill letter (e.g. a reason/instruction variable
  that failed to inject).

**`api/letters/generate/route.ts`**:
- `DISPUTE_SELECT` joins `dispute_reason:dispute_reasons(label)` and
  `dispute_instruction:dispute_instructions(label)`.
- `generateForDispute()` takes an optional `forceSource?: 'ai' |
  'agency_template'`. Effective source = `forceSource ?? dispute.letter_source`.
  - `'agency_template'` → `findBestTemplate(..., 'agency_static')` →
    `fillTemplateLetter()`. No match → error `"No matching agency template
    found for this item type."`
  - `'ai'` → existing flow, now requesting `kind: 'ai_prompt'` explicitly.
  - If `forceSource` differs from the dispute's stored `letter_source`, the
    same update call persists the corrected `letter_source` so later
    Regenerate clicks stay on the corrected path.

**`templates/actions.ts`**: `TemplateInput` gains `kind`;
`previewTemplateLetter()` branches to `fillTemplateLetter()` for
`agency_static` templates, passing sample reason/instruction labels so
those variables aren't blank in the preview.

## Round Workspace UX (`round-workspace.tsx`)

- Card headers (both "preparing" and "review" phases) get a small source
  badge (Sparkles = AI, FileText = Template) next to the existing
  bureau/letter-type badges.
- Bulk "Generate All" in the preparing phase runs each dispute through its
  own stored path transparently — no other change to that flow.
- In the review phase, a **failed** card whose `letter_source` is
  `'agency_template'` gets a secondary **"Generate with AI instead"**
  button next to Regenerate. Calls the generate endpoint with `forceSource:
  'ai'`, which also persists the corrected source.
- Successful cards never show the fallback button.
- PDF export and everything downstream reads `letter_content` exactly as
  today, regardless of source — no changes needed there.

## Out of Scope / Deferred

- No org-wide policy to force all letters onto one source — always a
  per-item choice.
- No manual per-template picker for the `agency_static` path — auto-matched
  via the same priority cascade as AI templates.
- No editing of custom reasons/instructions in place — delete and re-add.
- No filter tab for template kind on the Templates list.
